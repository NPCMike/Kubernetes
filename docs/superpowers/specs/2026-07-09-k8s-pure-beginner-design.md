# Kubernetes 純新手圖解闖關教材設計

## 目標

把現有 Kubernetes Killercoda 教材改成面向純新手小白的版本。學員不需要先知道 container、cluster、SRE 或 Kubernetes 名詞；教材要先用情境和圖解建立直覺，再用少量 `kubectl` 指令帶學員完成一次「觀察、判斷、修復」流程。

核心成果不是讓學員背名詞，而是讓他能回答三件事：

- 我現在在 Kubernetes 的哪一層看問題？
- 這個指令是在查什麼線索？
- 我看到這個狀態後，下一步該去哪裡找原因？

## 教材調性

保留 NPC 直播平台救援任務，改成更像新手入職第一天的互動任務。故事要有趣，但不能蓋過學習目標。

建議口吻：

- 像資深同事帶新人進戰情室，不像考試講義。
- 每個新名詞都先用生活比喻說明，再給正式名詞。
- 錯誤狀態不要寫得嚇人，要寫成「線索」。
- 每關結尾都要有一題小檢查，讓學員知道自己懂到哪裡。

避免口吻：

- 不用「你應該已經知道」這類預設背景的說法。
- 不一次塞太多架構名詞。
- 不把指令列成清單就結束，必須說明看哪一欄、代表什麼意思。

## 故事設定

學員是剛加入 NPC 直播平台的新手 SRE 實習生。坤哥直播帶貨造成流量暴增，平台開始不穩。學員接到的任務不是立刻重開機，而是跟著戰情室前輩一步步讀懂 Kubernetes 現場。

每一關是一個小任務：

- Intro：收到事故通知，知道自己要先觀察，不要亂修。
- Stage 0：拿到 Kubernetes 地圖，知道誰負責管理、誰負責跑服務。
- Stage 1：點名現場機器，分清控制中心與工作機器。
- Stage 2：讓一台工作機器暫停接新任務，理解維護節點。
- Stage 3：跟著線索查壞掉的 Pod，學會先看狀態，再看事件與 logs。
- Stage 4：刪掉一個 Pod，觀察 Deployment 自動補回，理解 desired state。
- Finish：用一張救援報告回顧自己已經會什麼。

## 每章固定結構

每個 Markdown stage 使用同一套教學節奏，降低新手認知負擔。

```md
# Stage X：章節標題

## 這一關的情境
用 2-4 句故事說明現在發生什麼事。

## 你先知道這個就好
用白話解釋本關 2-4 個核心概念。

## 看圖理解
放一張簡圖或引導讀現有圖片，指出這關要看哪個區塊。

## 跟著做
每個指令前先寫「這個指令要查什麼」。

## 看懂結果
放範例輸出，標出要看的欄位與判斷方式。

## 常見誤會
列出 1-3 個新手最可能誤解的地方。

## 小任務：確認你真的懂
用一題短問答或選擇題收斂理解。
```

## 圖文設計

圖不是裝飾，而是用來降低第一次理解的難度。每張圖都要回答一個明確問題。

現有三張圖的使用方式：

- `architecture.png`：放在 Stage 0 當總覽，但前面先加一段「先看左邊使用者、再看中間控制平面、最後看右邊工作節點」的導讀。
- `pod-debug-decision-tree.png`：放在 Stage 3 章末當查案小抄。正文先用四張狀態小卡拆解 `ImagePullBackOff`、`CrashLoopBackOff`、`Pending`、`Running 但不健康`。
- `deployment-reconciliation-loop.png`：放在 Stage 4，先用「外送訂單數量」或「值班人力補位」比喻 desired state 與 current state，再帶回圖中的左右兩側。

建議新增的輕量圖文元素：

- Intro：一張「救援任務地圖」，列出觀察 Node、排查 Pod、修復 Deployment 三站。
- Stage 1：`kubectl get nodes` 範例輸出表格，加上欄位註解。
- Stage 2：Cordon 前後對照圖，說明 Node 還 Ready，但不接新 Pod。
- Stage 3：四張錯誤狀態小卡，每張卡包含「看到什麼、代表什麼、先查哪裡」。

## 學習路線

教材要從具體到抽象：

1. 先說情境：直播平台不穩，不能亂重開。
2. 再看現場：有哪些 Node、哪些 Pod、狀態是什麼。
3. 再找線索：`describe` 看事件，`logs` 看程式輸出。
4. 最後抽象化：Kubernetes 一直把 current state 拉回 desired state。

這個順序要貫穿全文。不要一開始就長篇解釋 control plane、controller loop 或 reconciliation。

## 檢查理解的方式

每關都要有「小任務」，讓學員自己判斷是否理解。

範例：

- Stage 1：哪一台比較像指揮中心？你從哪個欄位看出來？
- Stage 2：`Ready,SchedulingDisabled` 是機器壞掉，還是暫停接新 Pod？
- Stage 3：看到 `CrashLoopBackOff` 時，哪個指令最接近應用程式錯誤原因？
- Stage 4：你刪掉的是 Pod，為什麼 Kubernetes 又補了一個新的 Pod？

小任務要能直接從本章內容回答，不要考本章沒有教過的知識。

## 範例改寫方向

原本：

> Cluster 是一組由 Kubernetes 管理的機器。

新手版：

> 你可以先把 Cluster 想成一整個直播後台機房。它不是一台電腦，而是一群被 Kubernetes 管理的機器。你不會直接問「哪一台主機壞了」，你會先問：「這個機房裡有哪些機器？誰負責管理？誰負責跑服務？」

原本：

> `STATUS` 是入口，不是結論。

新手版：

> `STATUS` 像急診分診牌：它會先告訴你病人目前看起來怎樣，但不會直接告訴你完整病因。看到異常狀態後，下一步要看 `describe`、`events` 或 `logs` 找線索。

## 驗收標準

- 純新手不需要先懂 Kubernetes，也能讀懂每章任務。
- 每個新名詞第一次出現都有白話解釋。
- 每個 `kubectl` 指令都有目的、範例輸出或觀察重點。
- 每章至少有一個圖、表格、狀態卡或視覺導讀。
- 每章都有一題能檢查理解的小任務。
- 不加入 Service、Ingress、RBAC、CNI、CSI 等超出本次入門範圍的主題。
- `k8s/index.json` 仍可被 JSON parser 正確解析。
- 所有圖片引用都指向存在的檔案，或以文字圖表方式呈現，避免破圖。

## 實作邊界

本次改寫只處理 Killercoda 教材文字與既有圖片導讀，不重做平台、不新增 Kubernetes 實驗環境，也不引入新的前端框架。

可以新增 Markdown 表格、ASCII 流程圖、簡單 Mermaid 圖或新的 PNG 圖片；若新增圖片，必須同步放入 `k8s/assets/` 並確認 Markdown 路徑正確。
